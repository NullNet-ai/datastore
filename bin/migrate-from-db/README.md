# migrate-from-db

Migrates data from a PostgreSQL database (**migrate_from**) into the store API (**migrate_to_store_url**) using the store’s create route.

1. **Login once** – POST to `/api/organizations/auth?app_id=portal` with credentials; receives a token.
2. **Read from Postgres** – For each table, runs `SELECT row_to_json(t) FROM "table" t` (raw SQL, no ORM).
3. **Insert into store** – For each row, POST to `/api/store/{table}?pluck=id,code` with `Authorization: Bearer <token>` and the row as JSON body. The same token is reused so the session is not re-created on every request.

Table order is read from the file generated by **order-tables** (`migrate_tables_order.rs`). Run `make order-tables` first.

**Running the migration:** Start the API server with `make run`, then trigger migration via **POST /migrate** with URLs and credentials in the JSON body. Use **GET /status** for progress and **GET /tables** for the table order and circular FK info. For one-shot from env (e.g. scripting), use `make run-once`.

**Source DB is read-only:** The migration only runs `SELECT` against `MIGRATE_FROM_DATABASE_URL`. The connection is set to `default_transaction_read_only = on`, so no INSERT/UPDATE/DELETE can run on the source (safe for QA). Writes go only to the store API (`MIGRATE_TO_STORE_URL`).

## Migration API (default: `make run`)

The default binary is an HTTP API server. Start it with `make run` (or `cargo run --bin migrate-from-db`). Listen port: **MIGRATE_API_PORT** (default `3090`).

| Route | Method | Description |
|-------|--------|-------------|
| **/migrate** | POST | Start migration. Body: `migrate_to_store_url`, `migrate_from_database_url`, `store_account_id`, `store_account_secret`, optional `store_root_account_id`, `store_root_account_secret`, `app_id`, `error_log_path`, optional **`tables`** (table order), **`circular_fk_cols`** (map of table → columns), and **`counter_service_url`**. When `counter_service_url` is set, rows from the **counters** table are sent to that service (POST /migrate) instead of the store. You must also provide **`migrate_to_database_name`** (e.g. `connectivo`, `skyll`) as the database name for Redis key prefixing; the API returns **400** if it is missing. Returns **202** if started, **400** if store/DB connection or login failed, **409** if a migration is already running. |
| **/generate-tables-order** | POST | Generate table order and circular FK columns **on the fly** from a live database. Body: `database_url`, optional `schema` (default `"public"`), optional `write_file` (default `false`), optional `output_path` (default `migrate_tables_order.rs`). Returns `{ tables, circular_fk_cols }`; if `write_file` is true, also writes the Rust file to `output_path`. Use this when you don't have a pre-generated file (e.g. in release mode). |
| **/status** | GET | Current migration state: `phase` (idle \| migration \| patch \| done \| error), `started_at`, `duration_secs`, `tables`, `table_stats` (per table: `records_total`, `records_inserted`, `errors_count`, `errors`), `patch_phase` (total_patches, patched_count, errors_count, `by_table` with patched/errors per table), `start_error` if start failed. |
| **/tables** | GET | Table order and circular FK columns from the **compiled-in** `migrate_tables_order.rs` (built at compile time). For a live-generated order, use POST **/generate-tables-order** instead. |

**Example (Postman / curl):**

```bash
# Start the API (from bin/migrate-from-db)
make run

# Trigger migration (admin + root account in body; optional counter_service_url for counters table)
curl -X POST http://localhost:3090/migrate -H "Content-Type: application/json" -d '{
  "migrate_to_store_url": "http://localhost:3002",
  "migrate_from_database_url": "postgres://user:pass@localhost:5432/sourcedb",
  "store_account_id": "admin@example.com",
  "store_account_secret": "your-secret",
  "store_root_account_id": "root@example.com",
  "store_root_account_secret": "root-secret",
  "counter_service_url": "http://counter-service:8080",
  "migrate_to_database_name": "connectivo"
}'

# Poll status
curl http://localhost:3090/status

# Table order and circular FKs
curl http://localhost:3090/tables
```

Before starting, the API validates store login (admin and root) and DB connection; if any fail, it returns **400** with the error message instead of starting.

**Generate table order on the fly (no pre-built file):**

```bash
# Generate order from source DB (returns JSON; optionally write .rs file)
curl -X POST http://localhost:3090/generate-tables-order -H "Content-Type: application/json" -d '{
  "database_url": "postgres://user:pass@localhost:5432/sourcedb",
  "schema": "public",
  "write_file": false
}'

# Then run migration using that order (paste tables + circular_fk_cols into the next request)
curl -X POST http://localhost:3090/migrate -H "Content-Type: application/json" -d '{
  "migrate_to_store_url": "http://localhost:3002",
  "migrate_from_database_url": "postgres://user:pass@localhost:5432/sourcedb",
  "store_account_id": "admin@example.com",
  "store_account_secret": "your-secret",
  "store_root_account_id": "root@example.com",
  "store_root_account_secret": "root-secret",
  "tables": ["accounts", "organizations", ...],
  "circular_fk_cols": { "account_organizations": ["account_id", "organization_id", ...], ... }
}'
```

### Release mode

In **release** (e.g. `cargo build --release` or a Docker image), the binary is built with whatever `migrate_tables_order.rs` existed at **compile time**. If that file was missing or from another DB, **GET /tables** and the default **POST /migrate** (without `tables` in the body) use that stale or empty order.

**Recommended in release:**

1. **Generate on the fly, then migrate with the result**  
   Call **POST /generate-tables-order** with the source `database_url` (and optional `schema`). Use the returned `tables` and `circular_fk_cols` in the next **POST /migrate** body. No file needs to exist on disk and no recompile is required.

2. **Optional: write the file**  
   If you want a physical `.rs` file (e.g. for a future build), set `write_file: true` and `output_path` in **POST /generate-tables-order**. The server will write `migrate_tables_order.rs` (or your path) on the host. The **current** process still uses the compiled-in order until you pass `tables` / `circular_fk_cols` in **POST /migrate** or restart after rebuilding with the new file.

So in release you can always: **POST /generate-tables-order** → then **POST /migrate** with the returned `tables` and `circular_fk_cols`. No `make order-tables` or pre-generated file is required at build time.

## Env (for one-shot: `make run-once`)

| Variable | Description |
|----------|-------------|
| `MIGRATE_FROM_DATABASE_URL` | Postgres connection string (source DB). |
| `MIGRATE_TO_STORE_URL` | Store base URL (e.g. `http://localhost:3002`). |
| `MIGRATE_STORE_ACCOUNT_ID` | Store login `account_id` (e.g. `admin@dnamicro.com`). |
| `MIGRATE_STORE_ACCOUNT_SECRET` | Store login `account_secret`. |

## Optional

| Variable | Description |
|----------|-------------|
| `MIGRATE_APP_ID` | Auth query param; default `portal`. |
| `MIGRATE_TABLES_FILE` | Path to the table-order file; default `migrate_tables_order.rs` (from order-tables). |
| `MIGRATE_BATCH_SIZE` | Number of rows to fetch from the source DB per batch (default `100`). |
| `MIGRATE_CONCURRENT` | Max concurrent requests to the store (or counter service) per batch (default `50`). **Must not exceed 100**; startup returns an error if it does. |
| `MIGRATE_ERROR_LOG` | File to append every error; default `migrate_errors.log`. Migration continues on row errors and exits non-zero if any were logged. |
| `MIGRATE_STORE_ROOT_ACCOUNT_ID` | Root account for **patch phase** (e.g. `root`). Used with `POST /api/organizations/auth?is_root=true`. If unset, the tool uses the normal account for root login and **patch requests will get 403 "Root access required"**. |
| `MIGRATE_STORE_ROOT_ACCOUNT_SECRET` | Root account secret (e.g. `pl3@s3ch@ng3m3!!`). Set together with `MIGRATE_STORE_ROOT_ACCOUNT_ID`. |
| `MIGRATE_METRICS_PORT` | If set (e.g. `9090`), exposes a Prometheus `/metrics` HTTP endpoint on this port for the duration of the migration. Use with Prometheus + Grafana for tables/records/errors/duration. |
| `MIGRATE_ONLY_PATCH_SECOND_PASS` | If `true`/`1`, skip the insert pass and only run the patch phase (collect patches from source DB, then PATCH). |

## Prometheus & Grafana

When `MIGRATE_METRICS_PORT` is set (e.g. `9090`), the migration binary serves **Prometheus text metrics** at `http://<host>:9090/metrics` while it runs. You can scrape this with Prometheus and build dashboards in Grafana.

**Metrics exposed:**

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `migration_rows_total` | Counter | `table`, `status` | Rows processed in insert pass; `status` is `success` or `error`. |
| `migration_rows_per_second` | Gauge | `table` | Approximate rows inserted per second per table (updated every 1s). |
| `migration_duration_seconds` | Histogram | `table` | Time spent migrating each table (insert pass). |
| `migration_fk_violations_total` | Counter | `table`, `constraint` | Foreign key constraint violations per table. |
| `migration_retry_total` | Counter | `table` | Retry attempts per table (increment when you add retry logic). |
| `migration_progress_percent` | Gauge | `table` | Progress per table (0–100). |
| `migration_second_pass_total` | Counter | `table`, `status` | Second pass (patch) results; `status` is `success` or `error`. |

**Docker Compose (recommended):**

From `bin/migrate-from-db`:

```bash
# Start Prometheus + Grafana
docker compose -f docker-compose.monitoring.yml up -d

# Run migration with metrics (in another terminal)
export MIGRATE_METRICS_PORT=9090
make run
```

- **Grafana:** http://localhost:3000 (login: `admin` / `admin`; change password on first login). Prometheus is pre-configured as the default data source.
- **Prometheus:** http://localhost:9091 (targets, graph). It scrapes the migration at `host.docker.internal:9090` while the migration is running.

**Manual setup:**

1. Run the migration with metrics: `export MIGRATE_METRICS_PORT=9090 && make run`
2. Point Prometheus at `localhost:9090` (see `prometheus.yml`).
3. In Grafana, add Prometheus as a data source and build panels from the metrics above (e.g. `migration_rows_total`, `migration_progress_percent`, `migration_second_pass_total`).

## Example

```bash
cd bin/migrate-from-db
make order-tables   # generate migrate_tables_order.rs from DB FK relations
make run            # start API on port 3090; then POST /migrate to run migration

```

For one-shot, ensure `migrate_tables_order.rs` exists in the current directory (or set `MIGRATE_TABLES_FILE`).

## order-tables (generate table order from DB)

A second binary in this crate connects to the database, reads **all tables and foreign-key relations** from `information_schema`, and **topologically sorts** them so that referenced tables come first. It writes the result as a **Rust array** to a file. The migration binary reads the table list from this file (no `MIGRATE_TABLES` env needed).

### Run via Makefile (from this folder)

```bash
cd bin/migrate-from-db
make order-tables
```

Uses `MIGRATE_FROM_DATABASE_URL` from `.env` or env. Writes `migrate_tables_order.rs` in this folder by default. Override with `MIGRATE_ORDER_OUTPUT=path/to/out.rs make order-tables`.

### Run directly

```bash
cd bin/migrate-from-db
export MIGRATE_FROM_DATABASE_URL="postgres://user:pass@localhost:5432/yourdb"
cargo run --bin order-tables
```

### Env

| Variable | Description |
|----------|-------------|
| `MIGRATE_FROM_DATABASE_URL` or `MIGRATE_ORDER_DATABASE_URL` | Postgres URL (required). |
| `MIGRATE_ORDER_SCHEMA` | Schema to scan; default `public`. |
| `MIGRATE_TO_DATABASE_URL` | **Required for unique indexes.** Postgres URL of the **destination** DB. Unique indexes are read from this DB and written into the generated file (so GET /tables and the migration can drop/recreate them). If unset, unique indexes come from the source DB (often empty). |
| `MIGRATE_ORDER_OUTPUT` | Output file path; default `migrate_tables_order.rs`. |

### Unique indexes (why GET /tables shows `unique_indexes: []`)

The generated file includes a `MIGRATE_UNIQUE_INDEXES` constant only when **unique indexes are read from the destination DB**. Do one of the following:

1. **From this folder, run the script** (sources `.env` so `MIGRATE_TO_DATABASE_URL` is set):
   ```bash
   cd bin/migrate-from-db
   ./run-order-tables.sh
   ```
2. **Or pass the destination URL on the CLI** (works from anywhere):
   ```bash
   cargo run -p migrate-from-db --bin order-tables -- --destination-url "postgres://admin:admin@localhost:5433/skyll"
   ```
3. **Or** ensure `MIGRATE_TO_DATABASE_URL` is in `bin/migrate-from-db/.env` and run `make order-tables` (the Makefile now sources `.env`).

Then **rebuild and restart** the API so GET /tables uses the new file: `cargo build -p migrate-from-db` and restart the server.

### Output

The script writes a Rust array, e.g.:

```rust
// Generated by migrate-from-db order-tables - do not edit by hand
// Tables in dependency order (referenced tables first). Use for MIGRATE_TABLES.

pub const MIGRATE_TABLES_ORDER: &[&str] = &[
    "organizations",
    "accounts",
    "samples",
    // ...
];
```

Order is strict: for every foreign key (A → B), table B appears before table A. If a cycle is detected, the script errors.
